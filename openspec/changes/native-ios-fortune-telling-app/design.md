## Context

当前工程是一个基础 SwiftUI iOS 应用，尚未形成业务页面、网络层、相机流程和本地数据模型。目标产品是纯原生 iOS 算命应用，命理内容由服务端大模型生成，客户端负责输入采集、权限处理、任务提交、状态展示、结果渲染和历史管理。

核心约束：
- iOS 端不在本地实现命理推理，不硬编码算命结论。
- 生日、面部、掌纹属于敏感数据，采集前必须给出清晰授权和用途提示。
- 掌纹/面相分析依赖图片上传，必须处理相机权限、相册导入、上传中、失败重试和删除。
- 服务端大模型结果必须结构化，客户端不能只依赖大段纯文本。

## Goals / Non-Goals

**Goals:**
- 建立 SwiftUI 原生应用的信息架构：首页、生日命理、掌纹面相、黄道吉日、分析结果、历史记录和设置。
- 建立统一 AI 分析客户端：提交请求、轮询或订阅任务状态、解析结果、展示错误。
- 建立三个核心业务流程：生日命理、掌纹/面相、黄道吉日。
- 建立隐私和历史能力：权限解释、本地历史、用户删除、上传数据删除请求。
- 让后续实现可以按 OpenSpec tasks 逐步落地，并能替换真实服务端接口。

**Non-Goals:**
- 不实现服务端大模型本身。
- 不在 iOS 端实现八字排盘、星座解析、掌纹识别或面相识别算法。
- 不在本次提案中定义订阅、支付、广告、登录体系。
- 不承诺命理结果的真实性或医学/法律/金融建议能力。

## Decisions

1. **使用 SwiftUI + 原生 MVVM 组织页面和状态**
   - 决策：用 SwiftUI 构建页面，用 `Observable`/`ObservableObject` 风格的 ViewModel 管理表单、加载、错误和结果状态。
   - 原因：当前工程已是 SwiftUI 入口，原生实现成本低，适合快速形成产品闭环。
   - 备选：UIKit 或混合架构。UIKit 对复杂相机控制更细，但当前核心是表单、结果页和轻量拍摄，SwiftUI 更一致。

2. **客户端只处理结构化输入输出，命理生成全部交给服务端**
   - 决策：iOS 端定义请求 DTO 和响应 DTO，例如 `AnalysisTask`, `BirthdayFortuneResult`, `PalmFaceReadingResult`, `AuspiciousDateResult`。
   - 原因：保证大模型提示词、命理规则和风控策略可在服务端迭代，客户端只随 schema 演进。
   - 备选：本地计算部分星座或八字。这样会引入规则一致性问题，也削弱“服务端大模型驱动”的边界。

3. **分析采用任务化协议**
   - 决策：提交分析后返回 `taskId`，客户端展示进度并通过轮询查询 `pending/running/succeeded/failed` 状态。
   - 原因：图片分析和大模型生成耗时不可控，任务化协议比同步等待更可靠。
   - 备选：单次同步请求。实现简单，但弱网和生成耗时下用户体验差，超时处理复杂。

4. **图片采集使用系统相机/照片能力，上传前提供预览与重拍**
   - 决策：掌纹和面相流程先使用原生相机/相册入口，用户确认后再上传。
   - 原因：减少误传敏感图片，方便后续加入裁剪、压缩和质量检测。
   - 备选：直接拍摄后上传。流程更短，但隐私风险和误操作成本更高。

5. **历史记录先本地保存摘要，服务端数据删除走显式请求**
   - 决策：本地保存分析类型、创建时间、输入摘要、结果摘要和服务端 `taskId`，用户删除时同时删除本地记录并向服务端发送删除请求。
   - 原因：兼顾离线复查体验和敏感数据删除诉求。
   - 备选：只保存服务端历史。需要登录/鉴权前置，本阶段不引入账号体系。

## Risks / Trade-offs

- [服务端 schema 未定] -> 先定义客户端协议抽象和 mock 数据源，真实接口确定后只替换实现层。
- [图片上传涉及敏感隐私] -> 上传前确认、结果页提供删除入口、设置页提供清空历史，并在 Info.plist 写清用途说明。
- [大模型结果不稳定] -> 要求服务端返回结构化字段和 disclaimer 字段，客户端对缺失字段做降级展示。
- [任务耗时较长] -> 采用任务状态页、轮询退避、取消入口和失败重试。
- [命理内容合规风险] -> 文案定位为娱乐参考，避免医疗、法律、投资等高风险结论。

## Migration Plan

1. 保留现有 SwiftUI 应用入口，新增页面目录、网络目录、模型目录和本地存储目录。
2. 先以 mock 服务实现三条业务闭环，确保页面、状态、历史和错误处理完整。
3. 接入真实服务端 API 地址、鉴权和返回 schema。
4. 补齐权限文案、隐私设置、删除接口和失败重试。
5. 上线前用 TestFlight 覆盖相机权限、弱网、图片上传失败、任务超时和历史删除。
